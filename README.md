# ТоргОС

Мульти-тенантная CRM/POS-платформа для малой розницы и общепита. Касса, учёт
товара, кабинет владельца с аналитикой из реальных чеков. Первый клиент —
семейный гастроном в Махачкале; архитектура с первого дня рассчитана на SaaS.

## Стек

- Next.js 16 (App Router) + TypeScript strict
- PostgreSQL + Prisma
- Tailwind CSS v4 (свои компоненты, без UI-китов)
- WebSocket (синхронизация остатков между кассами) — кастомный сервер `server.mjs`
- Auth — самописные сессии по паттерну Lucia (httpOnly-cookie, в БД только sha256 токена)

## Запуск

```bash
# 1. Postgres должен быть поднят; создать БД:
createdb redstore                      # или psql -c 'CREATE DATABASE redstore;'

# 2. .env: DATABASE_URL="postgresql://<user>@localhost:5432/redstore"

npm install
npx prisma migrate dev                 # схема + миграции
npm run db:seed                        # организация «Гастроном», 42 товара, 14 дней продаж

npm run dev                            # http://localhost:3000  (Next + WebSocket)
```

## Продакшн-развёртывание (Docker)

Всё, что нужно на сервере — Docker и Docker Compose.

```bash
cp .env.production.example .env
# отредактировать .env: POSTGRES_PASSWORD и AUTH_SECRET — сгенерировать:
#   openssl rand -base64 48

docker compose up -d --build
docker compose logs -f app        # убедиться, что миграции применились и сервер стартовал
curl http://localhost:3000/api/health
```

При первом запуске БД **пустая** — сид не запускается в проде автоматически
(нарочно: `db:seed` стирает все данные, в проде это опасно). Первый владелец
заводит организацию через `/register`. Если нужны демо-данные на новом
сервере — `docker compose exec app npx prisma db seed` (только на пустой БД).

Обновление после изменений в коде:
```bash
git pull
docker compose up -d --build app   # пересобрать и перезапустить только app;
                                    # entrypoint сам накатит новые миграции (migrate deploy)
```

### Защита данных: почему они не потеряются при баге

- **Данные живут в Docker-томе `pgdata`**, не в контейнере. `docker compose down`,
  падение приложения, пересборка образа — том не трогают. Тесты это подтвердили:
  контейнеры пересоздавались, данные оставались на месте.
- **Продажа — одна транзакция** (`server/services/pos.ts::commitSale`): при любой
  ошибке (нехватка остатка, обрыв связи, чужой товар) откатывается ВСЁ —
  «половинчатого» чека без списания или списания без чека не бывает. Покрыто
  тестами (`server/pos.test.ts`), включая намеренный сбой посреди чека.
- **История неудаляема из UI**: в схеме связи `Sale`/`StockMovement` → `User`,
  `Product` защищены от каскадного удаления — нельзя случайно снести историю
  продаж, удалив сотрудника или товар.
- **Ежедневный автобэкап** (сервис `backup` в compose): `pg_dump` в custom-формате
  в отдельный том `pgbackups`, хранится `BACKUP_KEEP` (по умолчанию 14) последних
  копий, старые ротируются автоматически.
- **Graceful shutdown**: `server.mjs` ловит SIGTERM/SIGINT (`docker stop`, деплой),
  закрывает WS-соединения и БД аккуратно, не рвёт запросы на середине.
- **Health-check** — `/api/health` проверяет живость БД; Docker перезапускает
  нездоровый контейнер автоматически (`restart: unless-stopped` + `HEALTHCHECK`).

### Восстановление из бэкапа

```bash
sh docker/restore.sh                              # покажет список бэкапов
sh docker/restore.sh redstore-20260717-030000.dump
```

Сценарий (потеря/порча данных → откат) проверен вручную: вставка записи → бэкап
→ намеренное удаление → restore → запись на месте.

### Бэкап на внешний диск/другой сервер

Том `pgbackups` — обычная директория Docker на диске сервера. Чтобы бэкапы
переживали потерю самого сервера, синхронизируйте её вовне, например cron на
хосте:
```bash
0 4 * * * rsync -a /var/lib/docker/volumes/redstore_pgbackups/_data/ user@backup-host:/backups/torgos/
```

## Архитектура (ключевое)

- **Tenant-изоляция в одном месте** — `server/tenant.ts`. `tenantDb(orgId)` —
  Prisma-расширение, автоматически фильтрующее каждый запрос по организации и
  проверяющее принадлежность на записи/создании. Сервисы не трогают `prisma`
  напрямую.
- **Продажа атомарна** — `server/services/pos.ts::commitSale`: Sale + SaleItem[]
  + списание остатка + StockMovement[] в одной транзакции. Деньги считаются на
  сервере (`Prisma.Decimal`), `priceAtSale` фиксируется на момент чека.
- **Рекомендации-правила** — `server/insights/index.ts`: контракт
  `generateInsights(input) → Insight[]`, честная статистика (нет продаж N дней,
  низкая маржа, кончится через X дней, истекает срок). Работает всегда, без
  сети и без ключей — источник истины, ИИ его не подменяет.
- **ИИ-сводка владельца** — `server/insights/ai.ts` + `server/ai/routerai.ts`:
  надстройка над теми же правилами через RouterAI (routerai.ru, OpenAI-совместимый
  API). Модели передают уже посчитанные из чеков агрегаты и сработавшие правила —
  она не считает и не выдумывает цифры, только расставляет приоритеты и
  формулирует текст. Кешируется в БД (`AiBriefing`, ключ — организация) на час,
  стримится отдельным `<Suspense>`-блоком, чтобы медленный ответ не блокировал
  остальной дашборд. Любая ошибка (нет ключа, таймаут, сеть) — тихая деградация
  до старого кеша или полное скрытие секции, без падения страницы.
- **Realtime** — `server.mjs` держит Next и WS в одном процессе; WS-соединение
  авторизуется cookie-сессией (storeId из БД, не от клиента), клиенты сгруппированы
  по точке. Мост к API-роутам — `server/realtime.ts` через общий `globalThis`.
